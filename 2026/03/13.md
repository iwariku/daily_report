 ##  わかったこと
### TypeScript

#### 役割の違い
#### 抽象クラス = 「共通の土台を持ったクラス」
抽象クラスは、
 - 共通のプロパティを持ってる
 - 共通のメソッドを持てる
 - 一部だけ実装を強制できる

#### interface + implements = 「形(契約)だけを定義」
interfaceは
 - 実装をもてない
 - プロパティやメソッドの「型」だけを定義する
 - クラスに「実装を強制」できる

#### できる事の違い
#### abstract(抽象クラス)
 - 実装を持つ事ができる
 - コンストラクタを定義できる
 - フィールド(プロパティ)を持てる
 - 通常のメソッドを定義できる
 - abstractメソッドで「実装の強制」ができる
 - 継承は1つだけ(単一継承)
 
#### interface
 - 実装は持てない(型定義のみ)
 - コンストラクタは定義できない
 - フィールドはもてない(値は持てない)
 - メソッドの「型」だけを定義できる
 - クラスに実装を強制できる
 - 複数implementsできる(多重実装可能)

#### 使い分けの基準

#### 抽象クラスを使う場面
 - 共通処理がある
 - ベースクラスとして機能させたい
 - 「is-a関係」が明確<br>例：
     - Animal ->  Dog / Cat
     - Food -> Apple / Pizza<br>

    のように明確な親子関係がある時に使う。

#### interfaceを使う場面
 - 実装はクラスごとにバラバラ
 - 共通の振る舞いだけ保証したい
 - 疎結合にしたい<br>例：
    - Logger
    - Repository
    - PaymentService<br>
「ConsoleLoggerはLoggerとして振る舞える」<br>
「FileLoggerもLoggerとして振る舞える」

ここでは「継承」ではなく、**「振る舞いの保証」** が目的

#### 設計思想

#### 抽象クラス = 継承ベース設計
 - 親のロジックを引き継ぐ
 - 依存がやや強くなる

#### interface = ポリモーフィズム設計
 - 振る舞いだけ統一。
 - 実装は完全に自由。
 - 依存を小さくできる。


#### 実務での判断基準
実務では基本的に
> 迷ったらinterfaceを使う

理由：
 - 疎結合にできる
 - DI(依存性注入)と相性がいい
 - テストしやすい
 - 拡張しやすい

抽象クラスは、**「本当に共通ロジックをまとめたいときだけ」** 使う
