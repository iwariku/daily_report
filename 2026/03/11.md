 ##  わかったこと

- `defer`は`tx(transaction)`が作られた直後に宣言した方が安全。

- `defer`は必ず呼び出される。<br>ただ、コミットが成功した後に、ロールバックが呼ばれてもデータベース側が「この件はコミットしているから、ロールバックする必要はないよ」と判断してくれる。

- ツイート一覧のように **「大量のデータを一度に取得する(:`many`)」** の場合は、以下の理由から`JOIN`が好まれる事があります。
  - **バッチ処理的な効率:**<br>100行取得する場合、サブクエリだと「100行✖️N個のサブクエリ」のクエリ発行が発生します。これを1回の大きな`JOIN`で結合してしまった方が、最適化しやすい場合がある。

  - **データの局所性:**<br>`GROUP BY`を使うことで、DBは「一旦ガバッと取ってきて、メモリ上で計算する」という動きをします。メモリが十分にある環境なら、この方が早く終わることがあります。

- `GetUser`1件取得と`GetTweets`一覧取得の違い<br>「1件だけ取る場合」と「100行取る場合」で最適な戦略が逆転することがある。
  - **ツイート一覧(`:many`)**<br>一覧の場合は、「一括処理」に向いているため、拡張性が高いと言える。

  - **ユーザー一覧(`:one`)**<br>詳細画面は「たった1行」を探す作業です。この場合、巨大な`follows`テーブルを`LEFT JOIN`してしまうと、1人のユーザーに数万のフォロワーがいた場合に**「中間データが数万に膨れ上がり、それを1に集約する」**という無駄な計算(CPU負荷)が発生してしまう。

 - サブクエリかJOINの判断は、基本的に **「取得したいレコード数(1件か大量か)」と「結合先のデータの多さ(1対1か1対多か)** のバランスで判断する。

- Discriminated Union・・・それぞれの状態を別々の方に定義し、共通のラベル(判別用のタグ)を持たせてから合体させる。
    ```
    // --- Discriminated Union ---

    // 1. 成功時の型
    type SuccessResponse = {
      status: 'ok';
      data: string;
    };

    // 2. エラー時の型
    type ErrorResponse = {
      status: 'error';
      code: number;
    };

    // 3. Union型にする
    type ApiResponse = SuccessResponse | ErrorResponse;
    ```
